# PR / MR 审查与修复循环

## 概述

本技能支持 3 类输入：本地仓库 diff、GitHub Pull Request、GitLab Merge Request。先识别审查来源，收集最小必要上下文，按优先级审查问题；如果发现问题，先列出问题和解决方案并等待用户确认；用户确认后，针对性修复并重新审查，直到没有阻塞性问题或用户停止。

默认使用中文输出。技术术语在英文更准确时可保留英文。

## 识别审查来源

将用户请求识别为以下三类之一：

- **本地 diff**：用户提到当前改动、已暂存改动、某次提交、分支对比，或直接提供补丁，但没有远程审查链接。
- **GitHub PR**：用户提供 GitHub Pull Request 链接、`owner/repo#123`，或明确提到 `PR` / `pull request`。
- **GitLab MR**：用户提供 GitLab Merge Request 链接、`group/project!123`，或明确提到 `MR` / `merge request`。

如果同时存在多种线索，优先采用最具体的一种。只有在审查范围选错风险较高时，才提出澄清问题。

## 总体工作流

1. **收集上下文**：读取变更文件列表、diff 摘要、关键 patch 和必要的邻近代码。
2. **执行审查**：按 P0、P1、P2 顺序查找安全性、正确性、性能、稳定性、规范性与测试问题。
3. **输出待修复报告**：如果发现问题，列出每个问题、影响、触发场景、解决方案和预计修改文件。
4. **等待用户确认**：不要在首次审查后直接修改代码。用户明确说“开始修复”“按方案修复”“继续修复”后，才进入修复阶段。
5. **针对性修复**：只修复当前报告中确认的问题。优先最小改动，遵守仓库的 `AGENTS.md` 和用户约束；不要擅自写删除操作代码。
6. **验证与复审**：运行与改动相关的测试、检查或构建；然后重新审查最新 diff。
7. **循环处理**：如果复审仍发现问题，再次输出待修复报告并等待用户确认。除非用户明确要求全自动循环，否则每轮修复前都等待确认。
8. **输出最终报告**：当复审通过时，输出最终版本结论和所有修改点总结。

## 修复阶段可用能力

修复代码时，可根据问题类型触发并使用相关 superpowers 技能：

- 遇到失败测试、异常行为或根因不清时，使用 `superpowers:systematic-debugging`。
- 实现功能性修复或补充测试时，使用 `superpowers:test-driven-development`。
- 改动较多、需要多步骤执行时，使用 `superpowers:writing-plans` 和 `superpowers:executing-plans`。
- 存在多个独立问题域时，使用 `superpowers:dispatching-parallel-agents` 拆分调查，但最终仍需统一复审。
- 声称完成前，使用 `superpowers:verification-before-completion` 做证据化验证。

使用这些能力时，仍必须服从当前仓库约束、用户确认边界和本技能的输出契约。

## 范围控制规则

- 优先关注变更代码，再根据需要查看周边上下文，以确认行为、约束条件或测试覆盖。
- 如果 diff 过大，优先审查核心逻辑文件，并明确标注此次审查是否为部分审查。
- 忽略 `vendor`、生成文件、lock 文件和纯机器格式化快照，除非用户明确要求审查，或其中包含有意义的手写逻辑。
- 将测试、配置、模式变更和接口契约视为一等审查对象。
- 修复阶段只改与已确认问题直接相关的文件；不要顺手重构无关代码。

## 收集审查上下文

### 本地 Diff 工作流

1. 先确定审查范围：未提交改动、已暂存改动、最近一次提交、指定提交区间，或指定补丁文件。
2. 先读取变更文件列表与 diff 摘要。
3. 只打开被修改文件，以及验证行为所必需的邻近代码。
4. 如存在相关测试则一并查看；如果高风险逻辑缺少测试，需要在待修复报告中明确指出。

### GitHub PR 工作流

1. 优先使用仓库宿主平台的官方工具或可用集成。
2. 收集 PR 标题、描述、base/head 分支、变更文件与 patch 内容。
3. 如果本地存在对应仓库，也检查本地已抓取的 PR diff、变更文件列表和 shortstat。
4. 在参考讨论区、标签或摘要之前，必须先审阅实际 diff。
5. 将 checks 与评论仅作为辅助上下文使用。

如果访问受阻，请向用户请求补丁、文件列表或可公开访问的链接。

### GitLab MR 工作流

1. 优先使用仓库宿主平台的官方工具或可用集成。
2. 收集 MR 标题、描述、源分支与目标分支、变更文件与 patch 内容。
3. 如果本地存在对应仓库，抓取或检查本地可见的 MR diff、变更文件列表和 shortstat。
4. 在参考讨论区或摘要之前，必须先审阅实际 diff。
5. 将流水线状态仅作为辅助上下文使用。

如果访问受阻，请向用户请求补丁、文件列表或可公开访问的链接。

## 内部审查清单

始终按以下优先级自上而下评估问题。不要在确认正确性和风险之前，把大量精力花在样式问题上。

### P0：安全性与正确性

重点检查：

- **资源泄漏**：数据库连接、文件、流、HTTP Response 是否正确关闭，例如 Go 中缺少 `defer resp.Body.Close()`。
- **并发安全**：共享状态是否在缺乏同步的情况下被访问，生命周期切换是否安全，是否存在 goroutine 协调问题。
- **错误处理**：是否忽略错误、吞掉失败、采用不安全兜底、出现部分写入成功路径，或缺少回滚 / 清理。
- **敏感信息**：是否硬编码 token、密码、私有地址、内网 IP，或把敏感信息泄漏进日志。
- **数据正确性**：是否存在无效状态流转、约束被破坏、nil 处理问题、索引越界、模式不匹配、单位错误或 API 假设不兼容。
- **逻辑漏洞**：是否存在明显的 logic flaw，例如条件反了、成功路径不可达、分支错误、状态流转缺失、兜底路径错误，或业务逻辑与预期行为冲突。
- **P0 映射规则**：已确认的安全问题、正确性缺陷与明显逻辑漏洞，默认按 P0 处理；除非上下文能明确证明影响较小。

### P1：性能与稳定性

重点检查：

- **重复查询**：循环或热路径中是否存在 DB、Redis、RPC 或文件系统重复调用。
- **内存分配**：是否存在不必要的大对象分配、重复拷贝、无界缓冲或可避免的序列化。
- **超时控制**：API、DB、RPC 或子进程调用是否缺少 `context`、超时或取消策略。
- **重试与背压**：是否存在无限重试、缺少抖动，或会对下游造成突发压力的模式。
- **运行稳定性**：外部调用、清理、重试与失败传播周围是否缺少必要的保护措施。

### P2：规范性与可读性

重点检查：

- **命名**：标识符是否符合项目约定，是否足够自解释。
- **代码异味**：函数是否过长（大约超过 50 行）、条件嵌套过深、职责混杂，或控制流难以理解。
- **冗余代码**：是否存在注释掉的代码、调试打印、死分支或复制粘贴块。
- **测试清晰度**：关键路径是否缺少测试，断言是否过弱，测试搭建是否脆弱。

## 项目上下文补丁

当仓库或请求明显属于某一特定领域时，在审查中补充领域专项检查。

例如：

- **媒体或图片处理项目**：检查 EXIF 解析、元数据保留、预览 / 缩略图兜底行为以及坏文件处理。
- **Flutter 项目**：检查 `build` 中是否有重计算、是否放大重建、controller 是否释放，以及 widget 销毁后是否仍异步更新状态。
- **Go HTTP 服务**：检查 response body 关闭、`context.Context` 传递、goroutine 泄漏、error wrapping 一致性，以及项目日志格式是否符合本地约束。

如果看不出明确的领域上下文，则只使用通用检查清单。

## 什么算一个 Finding

当某项变更有合理可能：

- 破坏行为或引入回归
- 错误处理非法输入或边界情况
- 违反接口、约束、模式或兼容性承诺
- 削弱安全、隐私或权限边界
- 引入并发、生命周期、清理或资源泄漏问题
- 明显损害性能或稳定性
- 让关键行为缺少测试保障

就应该提出 finding。

不要把纯粹的风格偏好升级为 finding，除非它掩盖了具体的维护性或正确性风险。

## 解决方案质量要求

每个问题都必须附带可执行的解决方案。

- 如果上下文足够，提供代码级修复片段或明确的修改步骤。
- 如果看不到完整实现，给出最小可信的补丁草图，并明确说明依赖的假设。
- 不要只给抽象建议，而不给下一步可执行动作。
- 修复方案要说明预计修改文件、验证方式和风险点。
- 多个问题相互独立时，按优先级分批修复；多个问题共享根因时，合并为同一修复方案。

## 输出模式

根据阶段和用户请求选择输出格式：

- **待修复审查阶段**：发现问题时，输出“待修复审查报告”，并停止等待用户确认。
- **修复执行阶段**：用户确认后，执行修复、验证和复审；如果复审仍有问题，再输出新的“待修复审查报告”。
- **最终通过阶段**：没有阻塞性问题时，输出“最终通过报告”，只总结结论、验证结果和所有修改点。
- **JSON 输出**：当用户明确要求 JSON、API 输出、机器可读输出或结构化数据时，仅返回严格 JSON，不加 Markdown 包裹。

## 待修复审查报告模板

发现任何 P0、P1 或值得修复的 P2 问题时，必须使用以下结构。输出后停止，不要直接改代码，除非用户在同一条请求中已明确授权自动修复。

```markdown
# Code Review: [MR/PR/diff 简述]

### 评审摘要
> **结论**: 建议修改 / 拒绝合并
> **变更规模**: 修改文件 X 个，新增 Y 行，删除 Z 行。
> **审查范围**: Local Diff / GitHub PR / GitLab MR
> **修复状态**: 等待用户确认后开始修复

---

### 本地修改内容
- **统计**: 修改文件 X 个，新增 Y 行，删除 Z 行。
- **关键文件**:
  - `path/to/file.go`: 说明该文件的核心改动。
- **差异摘要**: 用 1-5 条短句概括主要改动面。

### 待修复问题
1. **[P0/P1/P2 | 文件 | 行号] 问题标题**
   - **问题描述**: 说明具体问题。
   - **影响原因**: 为什么这是问题。
   - **触发场景**: 在什么情况下会出错。
   - **解决方案**: 给出具体修复方向。
   - **预计修改文件**: `path/to/file.go`
   - **验证方式**: 说明需要运行的测试、构建或手工验证。
   - **代码示例**:
```language
// 只放最小必要修复示例
```

### 建议修复顺序
1. 先修复 P0/P1 问题。
2. 再处理会影响可维护性或测试可信度的 P2 问题。

### 等待确认
请确认是否按以上方案开始修复。
```

## 最终通过报告模板

复审没有发现阻塞性问题时，必须使用以下结构。不要展示亮点、Open Questions / Assumptions、Testing Gaps / Residual Risk。

```markdown
# Code Review: [MR/PR/diff 简述]

### 最终结论
> **结论**: 通过
> **审查范围**: Local Diff / GitHub PR / GitLab MR
> **修复轮次**: X 轮

### 修改点总结
- `path/to/file.go`: 总结实际完成的修改。
- `path/to/test.go`: 总结补充或调整的测试。

### 验证结果
- `command`: 通过 / 未运行，并说明原因。

### 最终版本说明
当前版本已完成复审，未发现需要继续阻塞合并的问题。
```

## 结论状态判定规则

- 只有在复审没有阻塞性问题时，才能使用 `APPROVED` 或 `通过`。
- 当存在非平凡问题，但无需重构设计即可修复时，使用 `REQUEST_CHANGES` 或 `建议修改`。
- 当补丁引入了已确认的 P0 问题，尤其是严重安全问题、正确性缺陷或明显逻辑漏洞，且会直接影响核心行为或合并安全时，使用 `REJECT` 或 `拒绝合并`。
- 如果已确认存在安全问题、正确性 bug 或明显逻辑漏洞，则不得返回 `APPROVED`。
- 如果用户尚未确认修复，最终输出必须停在 `WAIT_FOR_USER_CONFIRMATION` 状态。

## 本地变更摘要规则

- 当本地 diff 证据与远程摘要同时存在时，优先采用本地 diff 证据。
- 只保留最相关的本地改动：文件统计、3-10 个关键文件，以及简短的高层摘要。
- 不要在回答中倾倒完整 patch，只需简明总结本地 MR / PR / diff 修改内容。
- 如果本地仓库不可访问，则 Markdown 中写 `无`，或 JSON 中将 `available` 设为 `false`。

## 证据规则

- 只要可用，就给出精确的文件路径和 1-based 行号。
- 只引用或转述支撑结论所需的最小内容。
- 当上下文不完整时，要把结论明确标记为假设，但最终通过报告中不要单独输出 Open Questions / Assumptions 栏目。
- 当 PR 或 MR 可能最近有更新时，在下结论前刷新远程审查数据。
- 修复后必须基于最新 diff 重新审查，不要只凭修复意图判断已通过。

## 触发示例

- `Use $pr-mr-review to review the current diff.`
- `Use $pr-mr-review to review these staged changes and list fixes before editing.`
- `Use $pr-mr-review to review PR 184 in owner/repo, then wait for approval before fixing.`
- `Use $pr-mr-review to repair this GitLab MR through review/fix/re-review cycles: https://gitlab.example.com/group/project/-/merge_requests/42`
